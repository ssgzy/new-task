## 事件 1：`Lion` 环境缺少 `datasets`
- 发现步骤：检查 `Lion` 环境可用依赖时，`conda run -n Lion python -c "import datasets, huggingface_hub, pandas, pyarrow"` 失败，报 `ModuleNotFoundError: No module named 'datasets'`。
- 影响：
  - 不能依赖 Hugging Face `datasets` API 直接重拉 FinQA。
  - 如果临时换环境或半途安装依赖，会让这轮新项目的可复现性变差。
- 解决方案：
  - 不切换环境，不临时装包。
  - 改用官方 GitHub 原始 `train/dev/test.json` + Python 标准库 `urllib` 重拉。
- 当前决策：数据重拉脚本 `scripts/prepare_finqa_data.py` 已固定为标准库下载实现。

## 事件 2：官方数据包含 `yes/no` 答案
- 发现步骤：第一次标准化后，validation `883` 条里有 `10` 条 `gold_numeric = null`；继续检查后确认这些样本并非脏数据，而是 `yes/no` 问题。
- 影响：
  - 本轮协议要求 numeric-only 输出。
  - 如果不处理，这些样本会在 `validation883`、`test1147` 中持续造成口径不一致。
- 解决方案：
  - 保留 `raw_answer` 原样。
  - 在 `gold_numeric` 中固定映射：`yes -> 1.0`、`no -> 0.0`。
  - 该规则写入 [[实验协议_v1]] 和 [[数据重拉与字段核对]]。
- 当前决策：标准化脚本已重跑，`train/validation/test` 的 `gold_numeric` 覆盖率现为 `100%`。

## 事件 3：`qa.answer` 与 `qa.exe_ans` 大量不一致
- 发现步骤：字段核对后统计到：
  - `train`：`4440/6251` 不一致
  - `validation`：`623/883` 不一致
  - `test`：`809/1147` 不一致
- 影响：
  - 不能把 `qa.answer` 和 `qa.exe_ans` 当成同义字段混着用。
  - 若 benchmark gold 口径漂移，会直接影响模型排名和后续 student selection。
- 主要原因：
  - 百分比缩放差异，例如 `93.5%` 对 `0.935`
  - 四舍五入差异，例如 `18.6` 对 `19.2`
- 解决方案：
  - `raw_answer` 优先固定取 `qa.answer`。
  - 仅在 `qa.answer` 缺失时回退到 `qa.exe_ans`，并记录 `source_answer_field_name`。
  - benchmark gold 统一以 `raw_answer` 的标准化结果为准，不用 `exe_ans` 覆盖。
- 当前决策：该规则已写入 [[实验协议_v1]]、[[数据重拉与字段核对]]，并落地到 `scripts/prepare_finqa_data.py`。

## 事件 4：`conda run` + heredoc 输出不稳定
- 发现步骤：使用 `conda run -n Lion python - <<'PY' ... PY` 时，部分探测命令出现空输出，需要额外轮询，影响排查效率。
- 影响：
  - 不直接影响 benchmark 结果，但会影响调试过程的可读性和稳定性。
- 解决方案：
  - 后续需要临时 Python 探测时，优先使用：
    - `conda run -n Lion python -c "..."`
    - 或 `/opt/homebrew/Caskroom/miniconda/base/envs/Lion/bin/python`
- 当前决策：探索命令和后续脚本执行均尽量避开 `conda run` 的 heredoc 形式。

## 事件 5：候选模型本地路径未齐全

  - 根据新增指令，直接执行 `scripts/ensure_candidate_models.py --download-missing` 补拉缺失模型到 Hugging Face cache。

## 事件 6：截断判定会低估 `truncation_without_answer_rate`
- 发现步骤：`Lion-7B` 的 1 条 smoke 在 `max_new_tokens=256` 下运行成功，但输出正好吃满 `256` 个新 token，且没有 `Answer:` 行。旧版 parser 只靠文本尾部启发式，未把这类情况标成 `truncated_suspect`。
- 影响：
  - 会低估长度校准里的 `truncation_without_answer_rate`
  - 可能错误放行本该被判为不安全的 `max_new_tokens` 档位
- 解决方案：
  - 将 `truncated_suspect` 规则升级为：`文本启发式 OR (命中 max_new_tokens 且缺失 Answer: 行)`。
  - 该信号由 runner 在 generation 后传入 parser。
- 当前决策：
  - `scripts/finqa_protocol_v1.py` 与 `scripts/run_finqa_local_benchmark_v1.py` 已按该规则修正。
  - 修正后将重跑 smoke，再进入正式 calibration。

## 事件 7：本机吞吐不足以今晚跑完 6 模型全流程
- 发现步骤：`Lion-7B` 的 1 条 smoke 在 `mps + float16 + max_new_tokens=256` 下，生成耗时约 `88.4s`，吞吐约 `2.89 tok/s`。
- 影响：
  - 若按同量级估算，本机上完成：
    - `6 模型 × 50 条 × 3 档` 的长度校准
    - `6 模型 × 200 条` 的准入筛选
    - 通过模型的 `883` 条完整 validation
  - 总耗时会到“几十小时”级，无法在今晚全部交齐。
- 提议解决方案：
  - 今晚先完成所有“可复现资产层”：数据、manifest、`protocol_v1`、模型注册表、runner、汇总脚本、模型下载。
  - 运行层改成“主表优先，appendix 后置”：
    - 先只跑主表 4 模型的 `val_calib50`
    - 再跑主表 4 模型的 `val_screen200`
    - 最后只对通过门槛的主表模型跑 `validation883`
  - 扩展表 2 模型转为补充阶段，不阻塞 student selection 主线。
- 当前状态：
  - 这是已确认的执行风险。
  - 用户已确认按“主表优先，appendix 后置”切换运行范围。

## 事件 8：长跑任务未增量落盘，存在中断即丢进度风险
- 发现步骤：主表长度校准启动后，`outputs/calibration_runs/Lion-7B/max_new_tokens_128/` 目录在运行过程中没有任何中间产物，说明旧版 runner 只有“全部跑完后一次性落盘”的行为。
- 影响：

- 解决方案：
  - `run_finqa_local_benchmark_v1.py` 改为逐样本追加写 `predictions.jsonl`
  - 增加 `--resume-existing`，读取已有 `predictions.jsonl` 跳过已完成样本
  - calibration / qualification orchestration 自动透传该参数
- 当前决策：
  - 已停止旧版主表校准
  - 正在切换到“增量落盘 + 断点续跑”版本后重启

## 事件 14：`chat_template` 不是四个主表模型的共同可用修复路径
- 发现步骤：
  - 按“输入接口正确性修复与验证”任务，运行 `scripts/debug_input_interface_v1.py` 对 4 个主表模型各做 1 条 smoke。

- 直接证据：
  - `Lion-7B`：`tokenizer.chat_template=False`，`auto` 实际渲染模式仍为 `plain`
  - `Orca-2-7B`：`tokenizer.chat_template=False`，`auto` 实际渲染模式仍为 `plain`
  - `MiniLLM-Llama-7B`：`tokenizer.chat_template=False`，`auto` 实际渲染模式仍为 `plain`
  - `Zephyr-7B-beta`：`tokenizer.chat_template=True`，`auto` 实际渲染模式为 `chat_template`，且 prompt 末尾有 `<|assistant|>`
- smoke 结果：
  - `Lion-7B`：`plain` 和 `auto` 两路都无 `Answer:`，且 `repetition_collapse=True`
  - `Orca-2-7B`：`plain` 和 `auto` 两路都无 `Answer:`，且 `repetition_collapse=True`
  - `Zephyr-7B-beta`：`plain` 下 `valid_parse=True`，但 `chat_template + repetition_penalty=1.1` 下 `Answer:` 后接解释文本，`valid_parse=False`
  - `MiniLLM-Llama-7B`：`auto` 仍为 plain fallback，有 `Answer:` 但答案行是算式，不是 parser v1 要求的纯数值
- 影响：
  - Claude 建议里的“只要补 `apply_chat_template()` fallback 就能修 Lion / Orca”在本地事实下不成立。
  - 当前不能把“chat template 缺失”直接当成导致 4 模型 calibration 崩溃的共同主因。
  - 如果现在直接重跑 `val_calib50`，大概率只是重复暴露 Lion / Orca / MiniLLM 的旧格式问题，还可能让 Zephyr 的 parser 合法性变差。
- 解决方案：
  - 暂停 canonical `val_calib50` 重跑。
  - 下一步先查 Lion / Orca / MiniLLM 的官方 instruction 包装格式，判断是否应引入“无 chat_template 时的统一非模板 wrapper 规则”。
  - 单独处理 special token 配置不一致问题后，再用 `outputs/debug/` 下的 1 条 smoke 验证。
- 当前决策：
  - 本轮只产出 provisional/debug 诊断，不进入全量 benchmark。
  - 不建议立刻重跑 `val_calib50`。

## 事件 32：Xwin-LM-7B 补跑 smoke 时 `raw_generation_text` 为空，底层 probe 确认模型连续生成 128 个 `<unk>`

- 发现步骤：
  - Xwin 补下载完成后，执行 `conda run -n Lion python scripts/debug_main_candidate_expansion_v1.py --labels Xwin-LM-7B --sample-index 0 --max-new-tokens 128 --repetition-penalty 1.1`
  - 检查 `outputs/debug/input_smoke_expansion/Xwin-LM-7B/smoke_expansion.json` 和 `raw_generation_registry.txt`
- 直接证据：
  - `smoke_expansion.json` 显示 `new_tokens=128`、`raw_generation_text=""`、`answer_appears=False`、`valid_parse=False`、`truncation_suspect=True`
  - `raw_generation_registry.txt` 是 0 字节
  - 进一步用 `conda run -n Lion python -c ...` 直接 probe 生成 token ids，得到：
    - `first_80_ids=[0, 0, 0, ...]`
    - `unique_id_count=1`
    - `decoded_skip_special_true=''`
    - `decoded_skip_special_false='<unk> <unk> ...'`
  - 该 tokenizer 里 `pad_token='<unk>'` 且 `pad_token_id=0`，所以 `decode(skip_special_tokens=True)` 会把这 128 个 `<unk>` 全部剥掉，最终看起来像“空输出”
- 影响：
  - Xwin 当前 Vicuna wrapper 下没有进入有效输入协议
  - 更准确地说，失败形态不是“正常文本里缺 `Answer:`”，而是“模型在当前输入/解码路径下直接退化成重复生成 `<unk>` token id 0”
  - 因此不能把 Xwin 当前 smoke 直接当作“模型已可比较但答错了”，它仍属于协议/生成链路未通过
- 解决方案：
  - 本轮先不把 Xwin 纳入正式 instruction 主表候选
  - 下一步如要继续排查 Xwin，应优先在 provisional 层检查“Vicuna serializer + 当前 tokenizer special-token 设置 + `pad_token_id=0`”是否存在组合性退化，而不是先改 parser 或直接重跑 calibration
- 当前决策：
  - Xwin 本轮 smoke 结论记为“未进入有效输入协议，生成退化为 `<unk>` 重复”
  - 不恢复 canonical 主线，不把 Xwin 计入已通过候选
