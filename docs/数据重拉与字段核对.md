# 数据重拉与字段核对

## 目标
- 重新拉取 FinQA 全量数据，不继承旧缓存和旧 split。
- 先核对原始字段，再做统一映射。
- 固定保留以下字段：
  - `id`
  - `pre_text`
  - `post_text`
  - `table`
  - `question`
  - `raw_answer`
  - `gold_numeric`
  - `split`
  - `source_answer_field_name`
  - `context_char_len`

## 执行原则
- 先打印列名，不默认相信旧脚本。
- 明确 `answer`、`answers`、`program` 等字段是否存在及其语义。
- 映射逻辑必须固化到脚本和 manifest 中，不允许 benchmark 时临时手改。

## 计划产物
- 原始 split 保存位置：
  - `data/raw/finqa/train.json`
  - `data/raw/finqa/dev.json`
  - `data/raw/finqa/test.json`
- 三个固定 manifest：
  - `data/manifests/val_calib50.jsonl`
  - `data/manifests/val_screen200.jsonl`
  - `data/manifests/validation883.jsonl`
- 字段核对元数据：
  - `outputs/metadata/finqa_split_summary.csv`
  - `outputs/metadata/finqa_manifest_index.csv`
  - `outputs/metadata/finqa_validation_field_summary.json`

## 标准化规则（当前冻结）
- `raw_answer`：优先取 `qa.answer`，缺失时回退到 `qa.exe_ans`
- `source_answer_field_name`：记录实际来源字段
- `gold_numeric`：
  - 普通数值：去掉 `$`、`,`、`%` 后解析
  - 带括号负数：转为负号再解析
  - `yes -> 1.0`
  - `no -> 0.0`
- `split`：官方 `dev` 统一映射为 `validation`
- `context_char_len`：按 prompt 使用的完整 context 串长度计算

## 当前结论
- 官方 split 大小已确认：
  - `train = 6251`
  - `validation(dev) = 883`
  - `test = 1147`
- `qa.answer` 与 `qa.exe_ans` 大量不一致，不能混用：
  - `train`：`4440/6251`
  - `validation`：`623/883`
  - `test`：`809/1147`
- 这些差异主要来自百分比缩放和四舍五入，因此 benchmark gold 必须以 `raw_answer` 标准化结果为准。
- validation 中 `871` 条来自 `qa.answer`，`12` 条因 `qa.answer` 缺失而回退到 `qa.exe_ans`。
- validation 中有 `492` 条百分比答案，`391` 条 plain 答案，其中 `yes/no` 共 `10` 条，已被标准化为数值。

## Manifest 规则
- `val_calib50`：validation 按 `seed=42`、按 `id` 排序后打乱取前 50 条。
- `val_screen200`：与 `val_calib50` 不重叠，取同一打乱序列中的后续 200 条。
- `validation883`：validation 全量 883 条标准化样本。
