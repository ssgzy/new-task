# 长度校准报告

## 目标
- 对每个候选模型在 `val_calib50` 上测试 `128 / 192 / 256` 三档。
- 输出并比较：
  - `answer_present_rate`
  - `valid_parse_rate`
  - `truncation_without_answer_rate`
  - `mean_new_tokens`
  - `p95_new_tokens`

## 冻结规则
- 选全体模型都安全的最小档。
- 安全定义：`truncation_without_answer_rate <= 2%`
- 今晚默认从 `256` 起步，待校准后决定是否降档。

## 当前发现
- `Lion-7B` 的 1 条 smoke 在 `max_new_tokens=256` 下：
  - `runtime_success = 1.0`
  - `format_ok = 0.0`
  - `valid_parse = 0.0`
  - `truncation_without_answer_rate = 1.0`
- 这说明不能把 `256` 直接视为“天然安全档”，必须等 `val_calib50` 的正式结果。

## 正式 calibration 结论
- `128` 档主表截断率：
  - `Lion-7B = 1.0`
  - `Orca-2-7B = 0.8`
  - `Zephyr-7B-beta = 0.32`
  - `MiniLLM-Llama-7B = 0.14`
- `192` 档主表截断率：
  - `Lion-7B = 1.0`
  - `Orca-2-7B = 0.78`
  - `Zephyr-7B-beta = 0.3`
  - `MiniLLM-Llama-7B = 0.12`
- `256` 档主表截断率：
  - `Lion-7B = 1.0`
  - `Orca-2-7B = 0.78`
  - `Zephyr-7B-beta = 0.2`
  - `MiniLLM-Llama-7B = 0.12`
- 按当前冻结规则：
  - 安全定义：`truncation_without_answer_rate <= 2%`
  - 结论：`128 / 192 / 256` 三档均不满足“全部主表模型都安全”
  - 因此当前没有可冻结的 `max_new_tokens`

## 运行策略
- 经确认，本轮先跑主表 4 模型。
- `scripts/run_length_calibration_v1.py` 现支持：
  - `--groups main`
  - `--labels "<模型名>"`
  - `--resume`
  - `--provisional`
- 这意味着：
  - 已完成的主表档位不会重算
  - 单模型或临时检查结果默认写到 `outputs/provisional/`
  - 正式主表聚合结果只写到 `outputs/calibration_report.csv`
