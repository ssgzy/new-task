- 总结已证实无效的旧尝试、当前 wrapper registry v2 设计、4 模型 smoke 结论，以及旧 debug 文件的归档位置。

## 不变的任务语义层
- context 相同
- question 相同
- direct-answer 要求相同
- decode config 相同
- parser 相同
- scoring 相同

## 允许变化的序列化层
- 按模型家族选择 wrapper serializer
- 不为单个模型改 parser
- 不为单个模型放宽 `Answer: <numeric value>` 约束

## v3 最小修复：Lion Alpaca wrapper + Orca slow tokenizer
- 本轮只做两项修复，不扩展研究问题、不改 parser、不改 screening 门槛、不重跑 canonical `val_calib50`：
  - Lion-7B 的默认 wrapper 从 `vicuna` 改成官方 Alpaca-style `### Instruction / ### Input / ### Response`
  - Orca-2-7B 的 tokenizer 加载显式固定 `use_fast=False`，保持 ChatML wrapper 不变
- Lion 当前字段映射：
  - `instruction` = 现有 `SYSTEM_PROMPT`
  - `input` = 现有 `build_user_prompt(context, question)`，因此 `### Context` / `### Question` 结构不变，只是被放入 Alpaca 的 `### Input:` 段
- Lion 目标序列化格式：
```text
### Instruction:
{SYSTEM_PROMPT}

### Input:
### Context
{context}

### Question
{question}

### Response:
```
- 官方依据：
  - [Lion 官方 train.py](https://github.com/YJiangcm/Lion/blob/master/src/train.py)
  - [Orca-2-7b 官方模型卡](https://huggingface.co/microsoft/Orca-2-7b/blob/main/README.md)
- 本轮新增实现偏差与处置：
  - 首次 `input_smoke_v3` v3a 中，Orca 虽然请求了 `use_fast=False`，但 `AutoTokenizer.from_pretrained(...)` 实际返回的是 `TokenizersBackend`
  - 同一条样本上，直接改用 `LlamaTokenizer.from_pretrained(...)` 后可输出 `Answer: 1.33%` 且 `valid_parse=True`
  - 因此已把 Orca 的 per-model tokenizer policy 从“只传 `use_fast=False`”升级为“显式指定 `LlamaTokenizer` 类 + 保留 `use_fast=False` 记录”，并将 v3a 旧输出归档后重跑
  - 注意：当前本地 Transformers 栈里 `LlamaTokenizer.is_fast` 仍显示 `True`，因此 Orca slow-tokenizer 修复验收以 `effective_tokenizer_class_name=LlamaTokenizer` 和 smoke 输出合法性为主，不再单看 `is_fast` 布尔值

## v3 后是否建议重跑 `val_calib50`
- 当前**不建议立刻重跑全主表 canonical `val_calib50`**。
- 理由：
  - Orca 已通过 1 条输入协议 smoke，值得进入 **单模型 provisional calibration**
  - 但 Lion 在官方 Alpaca-style 下仍然重复崩塌，说明“输入协议/模型快照/TokenizerBackend 组合”仍有未解问题
  - 现在直接恢复全主表 canonical calibration，仍会被 Lion 的输入协议失败拖住
- MiniLLM-Llama-7B 继续保持 `base_distilled` baseline 定位，本轮不再调 wrapper。

## 本轮确认“被修正的旧结论”
- Lion v2 `vicuna` 失败结论不能再直接外推为 Lion 模型能力结论：
  - 因为 Lion 官方训练模板是 Alpaca-style，不是 Vicuna-style
  - 所以 `Lion-7B -> vicuna` 的失败更可能是 serializer 选错造成的输入接口问题
- Orca v2 失败结论需要在 slow tokenizer 修复后重测：
  - 因为 Orca 官方模型卡明确要求 `use_fast=False`
  - 旧 v2 smoke 没固定这个 tokenizer policy，因此不能把失败直接解释成模型能力问题
- Orca v3a “已请求 slow tokenizer”结论也不能直接引用：
  - 因为本地 Transformers 栈里 `AutoTokenizer.from_pretrained(..., use_fast=False)` 对 Orca snapshot 实际返回的是 `TokenizersBackend`
  - 所以这次修复必须以“最终 tokenizer 类名 + 1 条 smoke 生成结果”作为验收证据，而不能只看 `requested_use_fast=False`

## 对 MiniLLM 的当前判断
- MiniLLM-Llama-7B 目前更适合保留为 **base distilled baseline**，不建议直接视为已经可进入统一 instruction benchmark 主表的 instruction-following 模型。
- 原因：
  - registry 已按 `completion` 包装进入 base distilled 路径
  - special-token pad/eos 已做最小对齐修复
  - 但 1 条 smoke 仍未产出 `Answer:` 行，`valid_parse=False`

## 是否建议立刻重跑 `val_calib50`
- 当前**不建议**。
- 理由：
  - 4 个主表模型里只有 Zephyr-7B-beta 在 registry v2 下达到 `Answer:=True` 且 `valid_parse=True`
  - Lion-7B / Orca-2-7B / MiniLLM-Llama-7B 仍未进入有效输入协议
  - 现在直接重跑 canonical `val_calib50`，大概率只是继续把输入协议问题投影成长度/截断问题

## 已证实无效的旧尝试

### 旧尝试 1：只依赖 `apply_chat_template` 的修复思路
- 为什么失败：
  - Lion-7B / Orca-2-7B / MiniLLM-Llama-7B 的 tokenizer 都没有 `chat_template`，`auto` 只能回退 plain prompt
  - 因此“一个 `apply_chat_template()` fallback 改动就能修 Lion/Orca”不成立
- 旧证据文件：
  - 原始位置：`outputs/debug/input_interface_diagnosis/`
  - 归档位置：`outputs/debug/archive/debug_failed_attempts/apply_chat_template_only_v1/input_interface_diagnosis/`

### 旧尝试 2：Zephyr 默认走 `chat_template`
- 为什么和 parser v1 冲突：
  - Zephyr 的 chat template 路径确实能生成 `<|assistant|>` 边界
  - 但在旧 smoke 中，`Answer:` 后面直接接了解释文本，例如 `Answer: To calculate ...`，导致 `valid_parse=False`
  - 这说明模板化 chat wrapper 和当前 numeric-only parser v1 并不天然兼容
- 旧证据文件：
  - 原始位置：`outputs/debug/input_smoke/Zephyr-7B-beta/smoke_compare.*`
  - 归档位置：`outputs/debug/archive/debug_failed_attempts/apply_chat_template_only_v1/input_smoke/Zephyr-7B-beta/`

### 旧尝试 3：plain fallback 直接喂 Lion / Orca
- 为什么不足：
  - Lion-7B 在旧 plain fallback 下无 `Answer:` 且重复崩塌
  - Orca-2-7B 在旧 plain fallback 下无 `Answer:` 且重复崩塌
  - 这说明“无 chat_template 就退回 plain”不是一个对所有 instruction 模型都足够的序列化策略
- 旧证据文件：
  - 原始位置：`outputs/debug/input_smoke/Lion-7B/smoke_compare.*`
  - 原始位置：`outputs/debug/input_smoke/Orca-2-7B/smoke_compare.*`
  - 归档位置：`outputs/debug/archive/debug_failed_attempts/apply_chat_template_only_v1/input_smoke/`

## 旧 debug 输出中仍有证据价值的内容
- `Lion-7B.txt/json`：证明 tokenizer 无 chat template、plain fallback 下重复崩塌、且 `model.config.pad_token_id` 与 tokenizer 不一致
- `Orca-2-7B.txt/json`：证明 tokenizer 无 chat template、plain fallback 下无答案行
- `Zephyr-7B-beta.txt/json`：证明 chat template 存在且能渲染 `<|assistant|>`，但旧 chat_template 输出不适配 parser v1
- `MiniLLM-Llama-7B.txt/json`：证明 completion/base distilled 模型不能简单等同 instruction model，且 pad_token_id 存在 config/tokenizer 不一致
