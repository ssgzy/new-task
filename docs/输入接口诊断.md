## 当前目标
1. 验证 chat template 是否正确应用
2. 验证 tokenizer / special tokens 是否匹配
3. 验证 prompt 是否按模型 instruction format 正确喂入

### 现状结论
- 当前 runner **不是无条件 plain prompt**。
- 当前逻辑是：
  - 若 `tokenizer.chat_template` 存在，则在 `auto` 模式下使用 `tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)`
  - 若 `tokenizer.chat_template` 不存在，则 fallback 到 plain prompt
- 因此本轮关键验证点不再是“代码有没有调用 chat template”，而是：
  - 哪些主表模型 tokenizer 实际有 `chat_template`
  - 没有 `chat_template` 的模型 fallback 后是否仍输出异常
  - `pad_token / eos_token / bos_token / unk_token` 与 `model.config` 是否存在明显错配

## Step 7：当前结论
- **chat template 缺失目前不能被证实为四个主表模型的共同主根因；wrapper registry v2 下，当前只有 Zephyr-7B-beta 已进入“有效输入协议”。**
- 更准确的结论是：
  - Zephyr-7B-beta 主线应继续保留 `plain` serializer，`chat_template` 只作为对照路径。
  - Lion-7B 用 `vicuna` wrapper 且修正 `pad_token_id` 后，仍然无 `Answer:` 并重复崩塌。
  - Orca-2-7B 用 `chatml` wrapper 且修正 `pad_token_id` 后，仍然无 `Answer:` 并疑似截断。
  - MiniLLM-Llama-7B 用 `completion` wrapper 且修正 `pad_token_id` 后，仍然无 `Answer:`；因此更适合保留为 `base_distilled` baseline，而不是直接纳入统一 instruction benchmark 主表。
- **当前不建议立刻重跑 `val_calib50`。**
- 原因：
  - 4 个主表模型里目前只有 Zephyr-7B-beta 的 `Answer:` 与 `valid_parse` 同时为 True。
  - Lion / Orca / MiniLLM 的问题还没有被 wrapper registry v2 完整解决，此时重跑 calibration 仍会把“输入协议问题”误投影为“长度问题”。

|---|---|---|
| Lion-7B | 官方训练模板是 Alpaca-style：`### Instruction:\n{instruction}\n\n### Input:\n{input}\n\n### Response:\n` 或 `### Instruction:\n{instruction}\n\n### Response:\n` | 当前给 Lion 配的 `vicuna` wrapper **很可能是错的**，应先切到 Alpaca-style 再做 1 条 smoke |
| Orca-2-7B | 官方模型卡明确给出 ChatML：`<|im_start|>system ... <|im_end|> ... <|im_start|>assistant`，并明确要求 `tokenizer = AutoTokenizer.from_pretrained(..., use_fast=False)` | 当前 `chatml` wrapper 方向是对的，但 **tokenizer 加载参数还没按官方要求固定 slow tokenizer** |
| MiniLLM-Llama-7B | 模型卡只说明训练数据来自 Dolly / Self-Instruct / Vicuna；代码侧 `generate.py` 直接读取 `prompt` 字段并 tokenization，模型卡没有提供单一固定 inference wrapper | 当前 `completion` wrapper 只能算 provisional baseline，不应假装它是 MiniLLM 官方唯一标准模板 |

## Step 9：新发现的影响结果问题
- Lion-7B：
  - 当前 wrapper registry v2 使用 `vicuna`
  - 但官方训练代码是 Alpaca-style `### Instruction / ### Input / ### Response`
  - 这意味着 Lion v2 smoke 的失败，不能直接解释为“模型不行”，而应先修正 serializer 再判定
- Orca-2-7B：
  - 当前 wrapper registry v2 的 ChatML 文本结构基本对齐官方卡片
  - 但 runner/debug 代码加载 Orca tokenizer 时默认没有显式 `use_fast=False`
  - 官方卡片明确指出 fast 和 slow tokenizer 会产生不同 token，因此当前 Orca smoke 结果仍存在“tokenizer 路径不完全符合官方推荐”的风险
- MiniLLM-Llama-7B：
  - 当前仍建议保留为 `base_distilled` baseline
  - 原因不是“已经找到官方 completion wrapper”，而是“官方资料本身没有给单一固定 inference wrapper，且 v2 smoke 仍未产生 `Answer:`”

## Step 10：建议的最小修复方案
1. Lion-7B：新增 `alpaca` serializer，把 Lion 默认 wrapper 从 `vicuna` 改成 `alpaca`，只在 `outputs/debug/` 下重跑 1 条 smoke。
2. Orca-2-7B：新增 per-model tokenizer load 规范，在 Orca 路径上显式 `use_fast=False`，保持 ChatML prompt 不变，只重跑 1 条 smoke。
3. MiniLLM-Llama-7B：暂不继续调 wrapper，不纳入“已进入有效输入协议”模型集合。
4. 在上述两项修复完成并通过 1 条 smoke 前，继续暂停 canonical `val_calib50`。

### v3 结论
- Lion-7B：
  - Alpaca-style prompt 已按官方训练模板生效
  - 但这条固定样本仍无 `Answer:`，且重复崩塌和疑似截断仍存在
  - 因此 Lion v2 的“vicuna 失败”结论不能再引用，但 v3 的“官方 Alpaca wrapper 下仍未进入有效输入协议”结论现在开始有解释价值
- Orca-2-7B：
  - ChatML wrapper 保持不变
  - 显式指定 `LlamaTokenizer` 类后，这条固定样本恢复到 `Answer: 1.33%` 且 `valid_parse=True`
  - 因此 Orca 旧失败更可能是 tokenizer 加载路径问题，而不是模型本身不会按协议输出
- 当前仍不建议立刻重跑全主表 canonical `val_calib50`；如果要继续，建议先只做 Orca 单模型 provisional `val_calib50`，Lion 继续单独排查。
- MiniLLM-Llama-7B 仍建议保持 `base_distilled` baseline 定位，本轮不继续调。
