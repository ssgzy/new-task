# 过程总结_主表扩展与新增模型筛查

## 文档目的
- 记录本轮“扩展正式主表候选池 + Orca 单模型 provisional calibration”的执行过程与结论。
- 只扩展候选池与输入接口诊断，不改变 FinQA direct-answer 主任务、不改 parser / EM / TM / FormatOK 规则、不引入 CoT 协议。
- 所有新增运行结果默认写入 `outputs/debug/` 或 `outputs/provisional/`，不覆盖 canonical `outputs/calibration_report.csv`。
- 关联文档：[[当前状态]]、[[异常与决策记录]]、[[任务笔记 - FinQA 基准重建]]、[[过程总结_输入包装修复]]、[[会话日志]]、[[模型注册表]]

## 为什么当前不能只靠 2 个模型构成正式主表
- 当前已确认 `Zephyr-7B-beta -> plain` 进入有效输入协议。
- 当前已确认 `Orca-2-7B -> chatml + LlamaTokenizer` 在 1 条固定样本上恢复 `Answer: 1.33%` 且 `valid_parse=True`。
- 但 `Lion-7B -> alpaca` 在官方 Alpaca-style wrapper 下仍 `Answer=False / valid_parse=False / repetition_collapse=True`。
- 因此如果只保留 Zephyr 和 Orca，正式 instruction 主表候选池过窄，不能支撑后续 student selection 对比。

## 为什么 Lion 单列而不是继续算主表名额
- Lion-7B 保留为“历史锚点 / 协议失败案例”，用于说明项目早期 adversarial distillation anchor 与当前协议边界。
- 但 Lion 在官方 Alpaca wrapper 下仍未进入有效输入协议，继续把它计入正式主表名额会把“协议失败”误混成“模型能力比较”。

## 为什么 MiniLLM 继续作为 base distilled baseline
- MiniLLM-Llama-7B 的官方资料没有给单一固定 inference wrapper。
- 当前 `completion` 路径下仍未稳定输出 parser v1 需要的 `Answer:` 数值行。
- 因此 MiniLLM 更适合继续作为 `base_distilled baseline` 对照组，而不是统一 instruction 主表模型。

## 为什么新增 4 个模型是合理的扩表方案
- `Xwin-LM-7B`：对齐类 7B instruction 模型，官方卡片给 Vicuna-style prompt，可作为 Zephyr/Orca 之外的新增 instruction 候选。
- `LaMini-LLaMA-7B`：理论上属于 instruction distillation 路线，但当前 HF API 下 `MBZUAI/LaMini-LLaMA-7B` 不可访问，因此本轮只保留占位，不下载、不 smoke。
- `DeepSeek-R1-Distill-Qwen-7B`：reasoning distillation 路线，可用于检验 reasoning-distilled 7B 在 direct-answer parser v1 下的协议可用性。
- `OpenR1-Distill-7B`：开源透明的 reasoning-oriented distill 路线，可与 DeepSeek 形成互补对照，但本轮仍不改 parser / 不引入 CoT。

## Orca provisional calibration 的目的
- 不宣布 Orca 最终“没问题”，而是验证 1 条 smoke 里的 `chatml + LlamaTokenizer` 修复是否能扩展到 `val_calib50` 的 50 条规模。
- 重点看：
  - `answer_present_rate`
  - `valid_parse_rate`
  - `truncation_without_answer_rate`
- 输出隔离到 `outputs/provisional/` 和新的 Orca calibration run 目录，避免复用旧 `AutoTokenizer` 版本 summary。

## 新增 4 模型首轮 smoke 结论
- `Xwin-LM-7B`
  - 首轮 registry status：`incomplete_local_snapshot`
  - 首轮结论：因热点暂停下载导致本地 cache 仍有 3 个 `.incomplete` blob，当时跳过 smoke
  - 当前状态：模型已补下载完成，registry 已刷新为 `available_local`，且已补跑 1 条 smoke
  - 补跑 smoke 结果：
    - `effective_wrapper=vicuna`
    - `tokenizer_effective_class=TokenizersBackend`
    - `answer_appears=False`
    - `valid_parse=False`
    - `truncation_suspect=True`
    - `parse_error_reason=missing_answer_line`
  - 关键补充诊断：
    - 虽然 `new_tokens=128`，但 `raw_generation_text` 为空串
    - 底层 token probe 显示 `generated_ids` 全是 `0`
    - 在该 tokenizer 中 `pad_token='<unk>'`、`pad_token_id=0`，因此 `decode(skip_special_tokens=True)` 会把全部 `<unk>` 剥掉
  - 当前结论：Xwin 当前失败形态是“Vicuna 路径下退化为连续 `<unk>` 输出”，仍未进入有效输入协议，不能加入正式 instruction 主表候选
- `LaMini-LLaMA-7B`
  - registry status：`pending_repo_id`
  - 结论：当前 `MBZUAI/LaMini-LLaMA-7B` 不可公开解析，本轮只保留占位，不下载、不 smoke
- `DeepSeek-R1-Distill-Qwen-7B`
  - tokenizer class：`TokenizersBackend`
  - effective wrapper：`chat_template`
  - smoke 结果：`Answer=False / valid_parse=False / truncation_suspect=True / repetition_collapse=False / parse_error_reason=missing_answer_line`
  - raw generation 前缀显示模型直接进入自然语言推理解释，没有收束到 parser v1 要求的最终 `Answer:` 行
- `OpenR1-Distill-7B`
  - tokenizer class：`Qwen2Tokenizer`
  - 推荐 wrapper：`plain`
  - smoke 结果：`Answer=True / valid_parse=False / truncation_suspect=False / repetition_collapse=False / parse_error_reason=invalid_numeric_literal`
  - 关键细节：`answer_line_raw` 是 `Answer: <numeric value>`，属于回显 prompt 占位符，不是合法数值答案

## Orca 256 档第三个 provisional summary
- `max_new_tokens=256`
- `answer_present_rate=0.86`
- `valid_parse_rate=0.78`
- `truncation_without_answer_rate=0.1`
- `mean_new_tokens=49.24`
- `p95_new_tokens=256`
- 当前解释：
  - `256` 档相对 `192` 档没有继续降低 `truncation_without_answer_rate`
  - 这说明剩余失败样本不是单纯“差一点 token 上限”，而是 Orca 仍倾向先输出较长自然语言推理，最终没收束到 `Answer:` 行
  - `256` 档下 5 条截断样本包括 `MSI/2006/page_61.pdf-1`、`CAT/2017/page_85.pdf-3`、`GIS/2012/page_64.pdf-1`、`MRO/2006/page_33.pdf-2`、`ILMN/2007/page_78.pdf-2`
  - 已进一步复查这 5 条样本的 prompt/output 结构，结论如下：
    - 5/5 都是 `prompt_wrapper_family=chatml`、`prompt_render_mode_effective=chatml`、`effective_tokenizer_class_name=LlamaTokenizer`
    - 5/5 的输出都以 `To calculate...` / `To answer this question...` 开头，先展开自然语言分步推理
    - 5/5 都吃满 `256` token，最终仍没有 `Answer:` 行，因此 `parse_error_reason=missing_answer_line`
    - 截断组 `prompt_length_tokens min/mean/max=958/1416.0/1633`
    - 非截断组 `prompt_length_tokens min/mean/max=543/1129.1/1736`
    - 非截断组最大 prompt 反而更长，所以这 5 条不太像“输入太长导致输出预算不足”，更像“这些样本触发了 Orca 在 direct-answer 协议下的长推理倾向”
  - 当前解释边界：
    - 这不是 parser bug，因为 parser 只是如实报告“没有最后答案行”
    - 也暂时看不出是统一 ChatML serializer / LlamaTokenizer 损坏，因为同一配置下 45/50 条不是截断，且其中 40 条直接以 `Answer:` 开头
    - 更合理的下一步是在 provisional 层单独试 `320/384`，或做一组“更强 answer-first 序列化收束”的对照，但不要立刻动 canonical 主协议
