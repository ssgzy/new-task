# 评分逻辑与报告写作指南

## 基本原则
- 不改 parser v1 的 strict 逻辑。
- 不改已经冻结的 `direct-answer` 设定。
- `strict scoring` 仍是主线。
- `relaxed scoring` 只作为补充视角，不替代 strict 结果。

## Strict vs Relaxed

### Strict scoring
- 仍然使用当前 `parser v1`：
  - 只认最后一条 `Answer:` 行
  - `Answer:` 后必须是合法纯数值字面量
  - 不能为单个模型写特殊 parser
- 当前主准确率指标定义为：
  - `Strict TM`

### Relaxed scoring
- 只做补充分析，不改变 strict 判定。
- 宽松提取策略：
  - 优先在 `Answer:` 行里提取
  - 若 `Answer:` 行带算式，则取最后一个 `=` 后的数值
  - 若没有 `Answer:` 行，则退回全文最后一个数值
- 作用：
  - 估计模型“算对了但没按 strict 协议落格式”的上界表现

### Gap 的含义
- `relaxed_gap_tm = relaxed_tm - strict_tm`
- 含义：
  - 被格式问题、回答落点问题、答案行序列化问题吃掉的 TM 准确率

## 为什么主准确率指标选 Strict TM，而不是 EM
- `EM` 太苛刻，只适合看完全一致的字面命中。
- FinQA 最终更关心数值是否正确，而不是小数位字符串是否一字不差。
- `Strict TM` 仍然要求：
  - 通过 strict parser
  - 落到合法 `Answer:` 数值行
  - 数值与 gold 在容差内匹配
- 因此 `Strict TM` 同时保留了：
  - 协议遵循
  - 数值正确性
  - 适合主表对比

## 指标解释

### Strict EM
- 定义：
  - strict parser 成功后，预测值与 gold 完全相等
- 用途：
  - 看“完全命中”的保守下界

### Strict TM
- 定义：
  - strict parser 成功后，预测值与 gold 在容差内匹配
- 用途：
  - 当前主准确率指标
  - 用于正式主表排序和比较

### Relaxed EM
- 定义：
  - relaxed 提取成功后，预测值与 gold 完全相等
- 用途：
  - 看“算对但 strict 格式不一定过”的宽松上界

### Relaxed TM
- 定义：
  - relaxed 提取成功后，预测值与 gold 在容差内匹配
- 用途：
  - 估计模型推理能力上界
  - 与 `Strict TM` 对比，用来衡量格式损失

### format_ok_rate
- 定义：
  - 输出中是否存在 `Answer:` 行
- 用途：
  - 看模型是否遵守最基本输出格式

### valid_parse_rate
- 定义：
  - strict parser 是否能把 `Answer:` 行解析成合法数值
- 用途：
  - 看模型是否不仅写了 `Answer:`，而且真的交出了可评分数值

### truncation_without_answer_rate
- 定义：
  - 输出疑似被截断，且直到结尾仍没有有效 `Answer:` 行
- 用途：
  - 看当前 `max_new_tokens` 是否足够安全

### avg_latency_ms
- 定义：
  - 单样本平均推理延迟
- 用途：
  - 比较实际运行效率

### tok_per_sec
- 定义：
  - 每秒生成 token 数
- 用途：
  - 看解码吞吐

### peak_vram
- 定义：
  - 单轮运行峰值显存 / 内存占用
- 用途：
  - 看部署成本和组员机器可复现性

### mean_output_tokens
- 定义：
  - 平均输出 token 数
- 用途：
  - 看模型回答长度与协议收束程度

## ChatGLM3 为什么叫边界通过候选
- `ChatGLM3-6B` 在 `screen200` 中：
  - `runtime_success = 1.000`
  - `format_ok = 0.830`
  - `truncation_without_answer_rate = 0.045`
  - 只有 `valid_parse = 0.595`
- 它不是正式通过，因为冻结门槛是 `valid_parse >= 0.60`。
- 但它只差 `0.005`，而且其余冻结指标都达标。
- 因此报告中更合适的叫法是：
  - `boundary-pass candidate`
  - 或 `边界失败 / 边界候选`
- 不建议把它直接写成正式主表通过模型。

## Zephyr 的完整故事
- 早期 smoke：
  - 单条样本看起来可用，似乎能进入 direct-answer 协议
- calibration：
  - 更大样本规模下暴露截断和格式问题
- screen200：
  - `format_ok = 0.430`
  - `valid_parse = 0.095`
  - `truncation_without_answer_rate = 0.105`
  - 明确淘汰
- 根本原因：
  - 在当前 `direct-answer` 协议下，`format compliance` 结构性不足
  - 不是单次 smoke 能反映出来的问题
- 报告中建议定位：
  - `DPO-only negative case`

## Qwen2.5 与 proposal 的关系
- 不要把 `Qwen2.5-7B-Instruct` 写成 distilled student。
- 更准确的表述是：
  - `aligned compact instruct model`
  - `strong aligned reference`
- 它的作用不是证明蒸馏本身，而是提供一个强的直接对齐紧凑模型参照。

## 建议的 Results 章节结构

### 1. 协议冻结与筛查
- 交代：
  - prompt / parser / decode config 已冻结
  - `screen200` 用于准入
  - `Strict TM` 为主准确率指标

### 2. 主结果表
- 只放正式主表通过模型
- 当前建议：
  - `Orca-2-7B`
  - `Mistral-7B-Instruct-v0.3`
  - `Qwen2.5-7B-Instruct`
  - `Yi-1.5-6B-Chat`

### 3. 效率表
- 单独列：
  - `avg_latency_ms`
  - `tok_per_sec`
  - `peak_vram`
  - `mean_output_tokens`

### 4. 边界与负例分析
- `ChatGLM3-6B`
  - 作为边界失败案例
- `Zephyr-7B-beta`
  - 作为 `DPO-only` negative case

### 5. 补充 relaxed scoring
- 在 appendix 或补充表中报告：
  - `Relaxed TM`
  - `Relaxed Gap`
- 用来说明：
  - 模型是没算对
  - 还是算对了但没按 strict 格式交卷

## 建议的表格格式

### 主结果表

  - 模型
  - base / 路线
  - Strict TM
  - Strict EM
  - FormatOK
  - ValidParse
  - Truncation

### 效率表
- 列建议：
  - 模型
  - avg_latency_ms
  - tok_per_sec
  - peak_vram
  - mean_output_tokens

### 负例 / 边界说明表
- 列建议：
  - 模型
  - 当前定位
  - 失败主因
  - 是否保留在 appendix
