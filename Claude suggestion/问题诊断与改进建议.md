3# FinBridge Benchmark 问题诊断与改进建议

> 写于 2026-04-02，基于对全部项目文件与 calibration 数据的完整阅读。

---

## 一、根本问题：Chat Template 未应用（最高优先级）

### 现象
- Lion-7B：`answer_present_rate = 0.0`，三档全截断，输出为 `"behaviour http « behaviour http «"` 无限循环
- Orca-2-7B：`answer_present_rate = 0.0`，三档均截断率 78-80%
- Zephyr-7B-beta：`answer_present_rate = 0.5`，但 `valid_parse = 0.24`，极低
- MiniLLM-Llama-7B：`valid_parse = 0.20`，勉强产出格式正确的答案

### 根源
`run_finqa_local_benchmark_v1.py` 第 205 行：
```python
prompt = build_prompt_from_record(record)
```
调用的是 `build_plain_prompt()`，生成的是纯文本字符串：
```
System:
You are a precise financial...

User:
### Context
...
```

这个字符串被直接喂给 tokenizer，**没有经过 `tokenizer.apply_chat_template()`**。

Lion-7B（Vicuna 架构）、Orca-2-7B（微软对齐格式）、Zephyr-7B-beta（`<|system|>`/`<|user|>`/`<|assistant|>` 格式）这三个模型都是 instruction-tuned 模型，它们在训练时的输入都经过了特定的 chat template 处理。不经过 template 直接输入纯文本，模型无法识别"自己该在哪里开始生成答案"，结果：

- Lion-7B 的 repetition collapse（生成重复 token 直到吃满上限）是经典的 **chat-template 缺失 + 无 repetition penalty** 的退化输出
- Orca-2-7B 也是同样原因产出大量无关内容直到截断
- Zephyr 部分能跟随，可能因为它的 eos token 设置不同

**这不是 max_new_tokens 的问题。把上限扩展到 512 不会解决 Lion-7B 的无限循环。**

### 修复方案（单处改动，影响全局）

在 `run_finqa_local_benchmark_v1.py` 的 `build_prompt_from_record` 调用后，改用如下逻辑：

```python
# 替换第 205 行
messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    {"role": "user", "content": build_user_prompt(context, question)},
]

if tokenizer.chat_template is not None:
    prompt = tokenizer.apply_chat_template(
        messages,
        tokenize=False,
        add_generation_prompt=True,
    )
else:
    # fallback for base models (e.g. MiniLLM-Llama-7B)
    prompt = build_plain_prompt(context, question)
```

同时在 generate 调用中加入：
```python
repetition_penalty=1.1,  # 防止 Lion-7B 类的 repetition collapse
```

### 预期效果
- Lion-7B 的 repetition collapse 会消失
- Orca-2-7B 的 answer_present_rate 应大幅回升
- Zephyr 的 valid_parse 应从 0.24 显著提升
- 整个 calibration 可以重新跑，结果才有意义

## 二、当前阻塞（长度冻结失败）是伪问题

### 现状
项目目前卡在：三档 `max_new_tokens`（128/192/256）没有任何一档能让所有模型满足 `truncation_without_answer_rate <= 2%`，因此主表 screening 暂停。

### 为什么这是伪问题
长度截断的根源是 chat template 缺失导致的输出退化，而不是 token 预算不够。修复 chat template 后，Lion-7B 和 Orca-2-7B 的输出长度会回到正常范围（数十 token），而不是当前的"吃满上限"。

### 建议策略
1. **先修复 chat template**，用当前的 `val_calib50` 重新校准，再做冻结决策
2. 如果修复后 Lion-7B 仍然表现很差（答案正确率极低），考虑将其作为"基线锚点"单列，不参与主表排名比较
3. **不需要扩展到 512**——在 chat template 修复后，256 对大多数模型应该足够

## 三、协议设计问题：禁止 Chain-of-Thought 会系统性压低准确率

### 当前协议
System prompt 明确要求：
```
Do not show your reasoning.
Do not explain.
Output exactly one final line in this format:
Answer: <numeric value>
```

### 问题
FinQA 的核心是多步数值推理（multi-step arithmetic）。对 7B 量级的小模型来说，**Chain-of-Thought（CoT）是它们能正确完成多步计算的必要条件**，而不是可选的附加项。

强制 direct-answer 格式在学术 benchmark 中有其合理性（结果可直接比较），但：
- 7B 模型没有 CoT 时，数值推理错误率会大幅上升
- 这会导致所有模型的得分系统性偏低，无法区分"真正理解了但格式没跟上"和"根本没推理能力"这两类失败
- 论文的核心 claim（distilled 小模型能否迁移到 finance）会因为协议设计缺陷而得出过度悲观的结论

### 建议
设计两个 prompt 变体同时运行：

| 变体 | 描述 | 用途 |
|------|------|------|
| `direct_v1` | 当前协议，不允许推理 | 测量 format compliance + immediate recall |
| `cot_v1` | 允许 CoT，最后一行仍为 `Answer: <value>` | 测量实际推理能力上限 |

这样可以量化"format constraint 造成的得分损失"与"推理能力本身的缺陷"，让 error analysis 更有说服力，论文的 contribution 也更清晰。

## 四、Gold 口径问题：`gold_numeric` 对百分比的处理有歧义

### 当前规则
`raw_answer` 取 `qa.answer`，然后去掉 `%`、`$`、`,` 再 parse 成 float。  
例如 `"1229%"` → `gold_numeric = 1229.0`

### 问题
smoke test 里看到有一条样本 `raw_answer = "1229%"`，`gold_numeric = 1229.0`。但 FinQA 原始数据中，百分比答案的实际值有两种表示：
- `qa.answer = "93.5%"` 意味着 93.5（百分比单位）
- `qa.exe_ans = "0.935"` 意味着 0.935（小数形式）

当前协议选 `qa.answer` 为准，去掉 `%` 后得到 `93.5`。但模型如果学会了 decimal 格式输出 `0.935`，与 gold `93.5` 的比较会失败（`valid_parse` 成功，但 EM/TM 失败）。

已知 validation 有 `623/883` 条在两个字段间存在数值不一致，主要就来自这种百分比缩放差异。

### 建议
在 TM（tolerance match）中增加一个百分比缩放豁免：
```python
def tolerance_match(pred, gold, rel_tol=0.02):
    if is_close(pred, gold, rel_tol):
        return True
    # 百分比缩放容忍：gold=93.5 pred=0.935 或反过来
    if is_close(pred * 100, gold, rel_tol) or is_close(pred, gold * 100, rel_tol):
        return True
    return False
```

这不会影响 EM，但会让 TM 更公平地反映模型的实际理解能力。

## 五、benchmark 设计的根本性建议：降低 screening 门槛的严格性

### 当前问题
准入筛选要求 `format_ok >= 70%` 且 `valid_parse >= 70%`，但在 chat template 问题修复之前，这些指标反映的是"prompt 格式是否被模型识别"，而不是"模型的金融推理能力"。

如果在 screening 阶段就淘汰掉 Lion-7B（因为格式问题），而论文的目标是研究"蒸馏小模型能否迁移到金融领域"，那么 Lion 这个具有代表性的 adversarial distillation 模型就被错误地排除在外了。

### 建议
- 先修复 chat template，再做 screening
- screening 的门槛应该反映"金融推理能力下限"，而不是"格式跟随能力的绝对门槛"
- 考虑将格式失败单独列为一列指标，而不是作为准入门槛

## 七、一句话总结

**当前项目的主要问题不是实验设计，而是实现细节——chat template 缺失导致全部 instruction-tuned 模型输出退化，这个一行代码的修复会让整个 calibration 阶段的数字从无意义变为有意义。在修复之前，任何基于当前 calibration 数据的决策都是在解一道不存在的题。**
